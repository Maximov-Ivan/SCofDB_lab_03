## 2. Найденные медленные запросы (до оптимизации)

### Запрос №1 Фильтрация по сумме с сортировкой по дате
1. SQL текста запроса:
```
SELECT id, user_id, total_amount, created_at, status
FROM orders
WHERE total_amount > 2000
ORDER BY created_at DESC
LIMIT 10;
```
2. План `EXPLAIN ANALYZE`:
- `Seq Scan` - полное сканирование всей таблицы
- `Filter` - проверка условия для каждой строки
- `Sort` - сортировка строк для поиска топ-10
3. `Execution Time`: 74.946 ms
4. Почему запрос медленный: полное сканирование таблицы, низкая селективность условия, сортировка большого набора данных

### Запрос №2 Фильтрация по статусу и диапазону дат
1. SQL текста запроса:
```
SELECT id, user_id, total_amount, created_at, status
FROM orders
WHERE status = 'paid'
  AND created_at >= '2024-07-01'
  AND created_at < '2025-01-01';
```
2. План `EXPLAIN ANALYZE`:
- `Seq Scan` - полное сканирование всей таблицы
- `Filter` - три условия проверяются для каждой строки
3. `Execution Time`: 29.674 ms
4. Почему запрос медленный: полное сканирование таблицы, нет индекса, который бы позволил быстро найти записи за нужный период

### Запрос №3
1. SQL текста запроса:
```
EXPLAIN (ANALYZE, BUFFERS)
SELECT
    o.user_id,
    COUNT(DISTINCT o.id) as orders_count,
    COUNT(oi.id) as items_count,
    SUM(oi.price * oi.quantity) as total_revenue,
    AVG(oi.price) as avg_item_price
FROM orders o
JOIN order_items oi ON oi.order_id = o.id
WHERE o.status != 'cancelled'
  AND created_at >= '2025-01-01'
  AND created_at < '2025-07-01'
GROUP BY o.user_id
ORDER BY total_revenue DESC
LIMIT 10;
```
2. План `EXPLAIN ANALYZE`:
- `Seq Scan on orders` - полное сканирование `order_items`
- `Seq Scan on order_items` - полное сканирование `order_items`
- `Hash Join` - соединяет `orders` и `order_items`
- `Sort` - сортировка строк перед группировной
- `GroupAggregate` - группировка пользователя
- `Sort` - сортировка строк для поиска топ-10
3. `Execution Time`: 487.523 ms
4. Почему запрос медленный: полное сканирование таблицы, нет индекса, который бы позволил быстро найти записи за нужный период

## 3. Добавленные индексы и обоснование типа
### Индекс №1
- SQL: `CREATE INDEX idx_orders_created_at ON orders USING BTREE (created_at);`
- Какой запрос ускоряет: Q1: запросы с фильтрацией или сортировкой по дате
- Почему выбран тип: BTREE оптимален для операций сравнения и сортировки

### Индекс №2
- SQL: `CREATE INDEX idx_orders_status_created_at ON orders USING BTREE (status, created_at);`
- Какой запрос ускоряет: Q2: фильтрация по диапазону дат и статусу
- Почему выбран тип: составной индекс BTREE фильтрует данные сначала по статусу, затем по диапазону дат

### Индекс №3
- SQL: `CREATE INDEX idx_orders_created_at_part ON orders USING BTREE (created_at) WHERE created_at >= '2025-01-01' AND created_at < '2025-07-01';`
- Какой запрос ускоряет: Q3: фильтрация по определенному диапазону дат
- Почему выбран тип: частичный индекс для определенного промежутка времени

RANGE партиционирование по кварталам:
- квартальные партиции оптимальны для текущего объема данных
- обеспечивают баланс между количеством партиций и размером каждой

### 5.2 Реализация
- создана пратиционная таблица `orders_partitioned`
- созданы партиции по кварталам и дефолтная партиция
- перенос данных
- создание индексов

### 5.3 Проверка эффекта

## 9. Выводы
- `EXPLAIN (ANALYZE, BUFFERS)` - главный инструмент диагностики
- Селективность - ключевой фактор эффективности индекса

- Сложные агрегации требуют структурных решений, а не только индексов
